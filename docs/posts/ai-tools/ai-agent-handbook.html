前言：為什麼要寫一本「使用手冊」

用 Claude Code 一段時間後，我累積了一大堆 skills——研究、寫作、投影片、數據分析、發筆記，零零總總幾十個。問題是：工具一多，反而不知道什麼時候該用哪個。更常見的狀況是，同一個 session 裡東問西問，最後 AI 開始「忘東忘西」、品質下滑。

於是我把自己的用法整理成一本手冊，回答三個問題：

  1. 我有哪些工具、什麼情況用哪個？
  2. 怎麼把零散的工具串成「工作流」，而不是每次重想？
  3. 怎麼有效率地用 AI agent——省 token、管理 context、寫好 CLAUDE.md、什麼任務該開哪家 AI？

這篇是那本手冊的公開版。內容裡所有關於 Claude Code 的事實，我都對照了 Anthropic 官方文件，沒有憑空編造（來源見文末）。

一、先建立正確的心智模型

AI agent ≠ 聊天機器人

聊天機器人是「你問、它答、它等你」。Agent（像 Claude Code）是「你描述目標，它自己讀檔、跑指令、改檔、驗證、迭代」——你在旁邊看、必要時導正，或直接走開 (Anthropic, 2026a)。

對研究的意義是：你不該再「自己想好每一步、一句句下指令」，而是把目標、限制、驗收標準講清楚，讓它自己跑。

比方說，不要說「幫我把這個變數 demean」，而是說：「用 first-difference 估這個 panel DML，跑完給我 bias / SE 表，並和 OLS 對照。」前者是把 agent 當手腳，後者才是把它當研究助理。

三個延伸機制：Skills / CLAUDE.md / Memory

機制 是什麼 角色
Skills（/skill-name 或自動觸發） 一包 SKILL.md ＋ 腳本，給 AI 特定領域知識與可重複流程 只在相關時才載入的「專門流程」
CLAUDE.md 每次開場自動載入的「專案說明書」 每次都載入的「常駐規則」
Memory 跨 session 的長期記憶（偏好、專案狀態） 你的「個人筆記」

一句話記住分工：CLAUDE.md＝每次都載入的常駐說明；Skill＝需要時才載入的專門流程；Memory＝你的偏好與狀態。三者別混用——這在後面省 token 的部分很關鍵。

二、把技能整理成「工作流」，而不是「工具清單」

工具清單會讓你每次都重想；工作流則是把「某個常見任務」固定成一條串接。我把自己的 skills 依場景分群，再串成幾條主力工作流。以下是幾個對研究生最實用的例子（觸發句直接貼給 agent 即可）。

工作流 A — 讀懂並報告一篇論文

串接： 讀 PDF → 建投影片 skill → 視覺／教學品管 → 上台

觸發句：「讀 papers/X.pdf，做一份詳盡的 Beamer 投影片，先建立『理解 → 大綱 → 逐張 frame → 版面』的中介表示，再寫 .tex，最後逐頁做視覺檢查。」

重點在「先理解、再規劃、才動手」——別讓 AI 從 PDF 直接跳到投影片，中間要有可檢查、可修補的階段。

工作流 B — 多篇文獻找研究缺口

串接： 文獻搜尋 skill → 研究缺口分析 skill → 把缺口磨成題目

觸發句：「分析 papers/ 這幾篇論文，做跨論文 synthesis（不要逐篇摘要），把 gap 分類並評 Strong / Medium / Weak，每個 claim 要標出論文出處，不確定的標 to verify。」

這條對寫 proposal、開題前掃描特別有用。關鍵是要求它橫向比較、證據落地、誠實標註不確定，而不是給你一堆樂觀的 bullet。

工作流 C — 從想法到論文初稿

串接： 訪談我 → 逼出 spec → 研究／寫作／審稿 pipeline → 自審

觸發句：「先用問答把我這個想法逼成一份 SPEC.md，確認後再走完整的研究 → 寫作 → 審稿 → 修訂流程，英文用我的口吻。」

「讓 AI 先訪談你」是官方很推薦的招：它會問你沒想到的取捨與 edge case，問完寫成 spec，再開新 session 執行——新 session context 乾淨、又有 spec 可循 (Anthropic, 2026a)。

工作流 D — 數據分析到出表／圖

觸發句：「跑這個 panel 的 FD-DML 與 OLS 對照，出 publication-ready 的 bias / RMSE 表和一張比較圖；data/raw/ 不要動，清理後存 data/processed/，跑完做 code review。」

要求它給可重現的腳本＋證據，而不是只丟一個數字。

共通原則：給它一個能跑的「驗收檢查」

不管哪條工作流，最有效的一招是：給 AI 一個能自己跑、會回傳 pass / fail 的檢查（測試、編譯、跑回歸對照、screenshot 比對）。有了它，AI 會自己跑到通過，你不用當「人肉驗證迴圈」(Anthropic, 2026a)。並且要求它給證據（測試輸出、指令回傳），而不是一句「我做好了」。

三、高效使用 AI agent 的核心：context 是最稀缺的資源

這是整本手冊最重要的一節。

為什麼 AI 用久了會「變笨」：context rot

AI 的 context window（脈絡視窗）裝著整段對話——每則訊息、每個被讀的檔、每次指令輸出。它會很快填滿，而且越滿、表現越差：模型開始「忘記」前面的指令、變多錯 (Anthropic, 2026a)。

Anthropic 把這現象叫 context rot：transformer 要處理 \(n^2\) 的 token 兩兩關係，context 越長、注意力被攤越薄；模型在超長序列上的訓練資料也較少。結果是準確率隨 context 變長而漸降（是斜坡、不是斷崖）(Anthropic, 2025)。

核心心法一句話：用「最小的高訊號 token 集合」達成任務，不是塞越多越好。

省 token 的具體做法

依重要性排序：

  1. 任務之間 /clear。換到不相關的工作就清空 context。最常見的浪費就是「kitchen sink session」（一個 session 裡東問西問）。
  2. 同一問題被你糾正超過兩次 → /clear 重開，用更好的 prompt。乾淨 session ＋ 好 prompt，幾乎總是贏過拖很長、塞滿失敗嘗試的 session。
  3. 階段做完就 /compact（不是等到變笨才壓）。可加指示：/compact 著重在 R 程式與回歸設定。
  4. 用 /context 看現在誰在吃 context。
  5. 要 AI「去讀檔」而不是「貼長文進來」。用 @檔名 引用、或叫它自己讀檔／抓網址。貼一大段論文進對話 = 永久佔 context；讓它讀檔 = 可控。
  6. 大量讀檔的探索 → 丟給 subagent。subagent 在獨立的 context 探索，只回報約 1,000–2,000 token 的摘要，不污染你的主對話 (Anthropic, 2025)。
  7. 小問題用 side question（/btw），答案不進對話歷史。
  8. CLAUDE.md 保持精簡（見下）——它每次都載入、整段常駐。

CLAUDE.md 是什麼、怎麼寫

定義： 每次對話開場 AI 都會自動讀的「專案說明書」。放它從程式碼推不出來、但每次都需要的東西：常用指令、程式風格、工作流規則、專案慣例 (Anthropic, 2026b)。

放的位置會自動合併：

位置 作用
~/.claude/CLAUDE.md 全域，套用到所有 session
./CLAUDE.md（專案根） 專案層，commit 進 git 與人共享
./CLAUDE.local.md 個人專案筆記，記得加進 .gitignore
父／子目錄 monorepo 會自動往上拉、讀子目錄檔時才拉子目錄的

官方的寫法守則很值得抄：

  • 保持短、人類可讀。每一行問自己：「拿掉這行會讓 AI 犯錯嗎？」不會就刪。
  • 臃腫的 CLAUDE.md 會讓 AI 忽略你真正的指令——重要規則淹沒在雜訊裡 (Anthropic, 2026a)。
  • 該放：猜不到的指令、與預設不同的風格、測試方式、repo 禮儀、專案特有的架構決定、環境怪癖、常見地雷。
  • 不該放：讀程式就知道的事、標準慣例、會頻繁變動的資訊、長篇教學。
  • 加 IMPORTANT / YOU MUST 可提高遵從度；在對話裡按 # 可把一句話直接寫進 CLAUDE.md；/init 可產生起始版本。

我自己的做法是：把專案根的 CLAUDE.md 當成路由表，只寫「哪個任務先讀哪個 STATUS.md」，把會變動的進度交給各專案的 STATUS.md。這樣常駐檔永遠不會塞會過期的內容——正好對應官方「會頻繁變動的資訊不要放 CLAUDE.md」這條。

先探索 → 再計畫 → 才動手

官方建議四階段：Explore（plan mode 只讀不改）→ Plan（產出計畫）→ Implement（寫並驗證）→ Commit。但 plan mode 有 overhead：小事（改錯字、加一行 log）直接做；只在「不確定做法、會動多個檔、對該段不熟」時才用 (Anthropic, 2026a)。一句話能描述改動，就跳過計畫。

四、不同 AI 的優勢與選用

模型版本與 benchmark 變很快，下面是 2026 年中的「相對強項」歸納，確切數字請臨用前再查 (Playcode Blog, 2026; Towards AI, 2026)：

模型家族 相對強項 什麼情況開它
Claude（Anthropic） 程式／agentic 任務、複雜邏輯與除錯、長流程、最不「AI 腔」的英文寫作 agentic 工作流、寫改 R／LaTeX、用自己口吻寫英文論文
GPT（OpenAI） 通用、知識面廣、結構化／商務型推理、外掛生態 快速廣博的解法、brainstorming、大量工具整合
Gemini（Google） 速度快、context window 超大、原生 Google 搜尋、多模態 一次塞超大量文字、需要即時網路資訊、處理圖片影片

對「計量 × 機器學習」這種研究工作流的實務建議：

  • agentic 流程 ＋ 寫 code ＋ 英文論文寫作（主力）→ Claude / Claude Code。
  • 超長一次性閱讀（一次丟十幾篇 PDF 做粗略 landscape）→ 可考慮 Gemini 的大 context；但要嚴謹 synthesis 仍建議走結構化的研究流程。
  • 快速查證／即時資訊 → 帶 Google 搜尋的 Gemini，或讓 Claude 用內建 web search。
  • 不要迷信單一 benchmark 冠軍：挑模型看「你的任務類型」，不是看排行榜。

至於 Claude 家族內部：Opus 做最難的推理與長流程、Sonnet 是又快又強的日常主力、Haiku 處理量大重複的小事；趕時間又想要 Opus 品質可開 fast mode。簡單任務用小模型、難任務才上 Opus，是最直接的省錢心法。

小結

把這套方法濃縮成一張表：

主題 一句話心法
心智模型 描述目標與驗收標準，讓 agent 自己跑，別當它的手腳
工作流 把常見任務固定成串接，附「觸發句」，每次複製即用
驗收 給一個能自己跑、回傳 pass/fail 的檢查；要證據不要「我做好了」
context 它是最稀缺資源；越滿越笨（context rot）
省 token 任務間 /clear、階段後 /compact、讀檔別貼長文、探索丟 subagent
CLAUDE.md 短、只放猜不到又每次要的；會變動的進度交給 STATUS.md
選 AI 看任務類型選家族；Claude 做 agentic／寫作、Gemini 大 context、GPT 廣博

工具會一直變多，但「先理解、再規劃、才動手；管好 context；給它能驗證的檢查」這幾條原則不太會過期。這份手冊我會持續更新——用一陣子後，把真正好用的工作流沉澱下來，比追新工具更重要。
